FULL-STACK PROJECT

LLM-powered Medical Care Plan Generation

Replacing a 20-40 minute manual pharmacist workflow with a structured, auditable generation pipeline

Duration Feb 2026 – Present
Stack Django + DRF
Infrastructure Docker / AWS-ready
Status Active Development

Motivation

Pharmacists spend 20-40 minutes manually drafting care plans for each patient: reviewing medication orders, checking interactions, writing recommendations. This project replaces that workflow with a structured pipeline: validated intake, rule-based checks, LLM drafting, human review, and export. The system prioritizes correctness and traceability over free-form chat.

Architecture Evolution

Over a 16-day development sprint, the architecture evolved from a monolithic Django app to a production-ready microservice system:

Phase 1: Monolith (Days 1-4)

  • Single Django application with synchronous LLM calls
  • SQLite database, direct API integration
  • Problem: LLM calls blocked the request thread (30-60 seconds)

Phase 2: Task Queue (Days 5-9)

  • Added Celery + Redis for async LLM processing
  • PostgreSQL migration for production reliability
  • Status-driven lifecycle tracking (PENDING, PROCESSING, COMPLETED, FAILED)
  • Retry/backoff on API failures

Phase 3: Production Architecture (Days 10-16)

  • Adapter-style ingestion for multiple order sources (HL7, CSV, manual)
  • Business-rule duplicate detection before LLM invocation
  • Dockerized services (PostgreSQL + API + Worker containers)
  • Structured for AWS deployment (ECS/Fargate)

Technical Deep Dive

API Design

Django REST Framework with strict schema validation and idempotent request handling. Key endpoints:

  • POST /api/orders/: Create medication order (validated, deduplicated)
  • POST /api/care-plans/generate/: Trigger async care plan generation
  • GET /api/care-plans/{id}/: Retrieve care plan with status
  • GET /api/care-plans/{id}/audit/: Full audit trail

Error Hierarchy

Custom exception classes mapped to HTTP status codes, with structured error responses that include field-level validation errors, business rule violations, and system errors separately.

Data Model

Model Fields Purpose
MedicationOrder patient_id, drug, dose, route, frequency, source Incoming order from pharmacy system
CarePlan order_id, status, content, llm_model, prompt_version Generated care plan with lifecycle tracking
AuditLog care_plan_id, action, timestamp, actor Every state transition is logged
